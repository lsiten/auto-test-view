## 方案

### PageIndex 策略（方案 C：单服务单 workspace，doc-map 标注 scope）

**核心原则**：Python 服务零改动，所有 scope 逻辑在 TypeScript 侧处理。

- PageIndex 服务保持一个进程、一个 workspace，统一放在 `~/.auto-test-view/pageindex/`
- Markdown 文档统一放在 `~/.auto-test-view/docs/`，文件名含 scope 前缀：`project_rec-xxx.md` / `global_rec-xxx.md`
- doc-map.json 结构升级：
  ```json
  {
    "rec-20260401-001": { "docId": "doc-aaa", "scope": "project" },
    "rec-20260401-002": { "docId": "doc-bbb", "scope": "global" }
  }
  ```

### semantic-index.ts 改造

1. **路径统一**：
   - `getDocsDir()` → `~/.auto-test-view/docs/`（不再按 scope 分目录）
   - `getWorkspaceDir()` → `~/.auto-test-view/pageindex/`（同上）

2. **doc-map 升级**：
   - 类型从 `Record<string, string>` 改为 `Record<string, { docId: string; scope: RecordingScope }>`
   - `getDocMap(scope?)` — 无 scope 返回全部，有 scope 则过滤

3. **indexRecording(recording, config, scope)**：
   - Markdown 文件名：`${scope}_${recording.id}.md`
   - doc-map 写入时记录 scope

4. **removeProfile(recordingId, scope?)**：
   - 从 doc-map 查找对应条目（带 scope 匹配）
   - 删除对应 Markdown 文件和服务端文档

5. **rebuildAllProfiles(config)**：
   - 从 store 的 `listRecordings()` 获取全部录制（已含 scope）
   - 按各录制的 scope 重新索引

6. **getDocumentList(scope?)**：
   - 从服务获取全部文档，按 doc-map 中的 scope 过滤

7. **getIndexStatus(scope?)**：
   - 按 scope 过滤统计

### matcher.ts 改造

`matchRecording(task, currentUrl, config)` 改为两轮查询：

```
1. Phase 1a: 只用 project scope 的文档列表 → LLM 选择
   - 匹配成功（confidence >= 0.75）→ Phase 2/3 验证 → 返回
2. Phase 1b: 只用 global scope 的文档列表 → LLM 选择
   - 匹配成功 → Phase 2/3 验证 → 返回
3. 都没匹配 → 返回 noMatch
```

`replayRecording` 不变（已有完整 Recording 对象，不关心 scope）。

## 验收标准
1. Python 服务零改动，单进程单 workspace
2. doc-map.json 存储 scope 信息
3. Markdown 文件名含 scope 前缀，统一在 `~/.auto-test-view/docs/`
4. `indexRecording` 正确按 scope 写入
5. `getDocumentList(scope)` 按 scope 过滤返回
6. matcher 先查 project 文档，无匹配再查 global
7. TypeScript 编译通过
