# 失败原因分析规则

当测试用例执行失败时，必须进行原因分析。

## 单元/集成测试失败

1. 读取错误堆栈，定位具体报错行
2. 读取对应源码文件的相关代码段
3. 分析期望值与实际值的差异
4. 推理可能的原因（逻辑错误/边界遗漏/mock 不匹配/接口变更）
5. 给出修复建议

## E2E 浏览器测试失败

1. 读取 MCP 工具返回的错误信息
2. 如果是元素未找到：检查 index 是否正确、页面是否加载完成
3. 如果是超时：分析页面加载时间、异步操作
4. 如果是断言失败：对比实际 URL/DOM 与期望值
5. 截图保存 → 辅助人工排查
6. 给出修复建议（调整等待时间/更换定位方式/修改断言）

## 失败报告格式（强制）

每条失败用例在 report.md 中必须包含以下全部字段：

```markdown
### FAIL: {caseId} {name}
- **类型**: unit | e2e | integration
- **文件**: 测试文件路径
- **错误信息**: 原始错误消息
- **失败原因**: 根因分析（不能只复述错误信息，要分析为什么）
- **代码位置**: 相关源码位置（file:line）
- **修复建议**: 具体可操作的修复方向
```
